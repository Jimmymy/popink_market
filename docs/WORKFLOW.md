# Popink Market 工作规则

本文档定义项目中的规划、实现、文档、测试、验收和 Lark 同步如何保持一致。

## 事实来源

- `SPEC.md` 定义产品范围、业务规则、测试环境标准、生产前阻塞项和验收标准。
- `docs/MILESTONES.md` 定义执行顺序、任务状态和进度日志。
- `README.md` 定义运行、验证、测试和交付约定。
- `AGENT.md` 定义工作区操作手册和后续代理约束。
- 代码与配置定义真实实现状态。

当这些内容不一致时，按下面顺序修正：

1. 产品范围、生产目标、支付物流策略或测试门禁变化，先改 `SPEC.md`。
2. 任务状态、里程碑或日志变化，改 `docs/MILESTONES.md`。
3. 启动命令、路径、账号、同步方式或操作说明变化，改 `README.md` 与相关 `docs/`。
4. 最后再补代码、配置和验证，使实现与文档一致。

## 中文文档规则

- 项目中的说明文档统一使用中文。
- 标题、正文、备注、步骤和规则说明都应优先使用中文。
- 允许保留英文的范围仅包括：代码、命令、文件路径、环境变量、URL、第三方固定名称和必要术语。
- 新增文档脚本时，其说明文档必须同步提供中文版本。
- 如在 PowerShell 中查看中文文档，优先使用 `npm run docs:read:utf8 -- <path>`。

## 生产目标原则

- 项目始终以上线生产为目标推进。
- 当前最低交付标准是 `staging / 测试环境` 可运行、可验证、可迁移。
- 本地运行只作为开发、排错和提交前验证手段。
- 任何“临时模拟能力”都必须标注适用范围和生产前替换条件。
- 支付和物流是结构稳定后的第一优先级，不再作为远期预留项。

## 里程碑工作流

开始一个有意义的实现任务前：

- 确认任务属于哪个里程碑。
- 查看 `docs/MILESTONES.md` 中关联的 `SPEC.md` 章节。
- 如果任务已被跟踪，把相应状态标记为 `In Progress`。
- 明确该任务是否影响测试环境、生产前阻塞项、支付物流或测试门禁。

实现过程中：

- 改动尽量限制在当前里程碑范围内。
- 出现新需求时，先改 `SPEC.md`，或先记录到里程碑的待确认项，再进入代码实现。
- 涉及支付、物流、环境变量、部署或安全时，同步更新对应文档。
- 若任务延期，标记为 `Deferred` 并说明原因。

实现完成后：

- 运行最小但有意义的验证命令。
- 新增或修改功能必须同步补单元测试。
- 更新 `docs/MILESTONES.md` 中的任务状态和进度日志。
- 如果命令、路径、内容页、脚本、支付物流或同步方式变化，更新 `README.md` 和相关说明。

## 测试门禁

交付采用“关键模块 + 覆盖率”标准：

- 核心业务模块必须逐步补齐单元测试：前台购买链路、账号注册登录、配置/env、支付、物流、文档同步脚本。
- 新增功能必须随代码一起提交单元测试。
- 修改既有核心模块时，必须补齐被触碰行为的测试。
- 支付相关改动必须覆盖成功、失败、取消、回调或 fallback 中被触碰的路径。
- 物流相关改动必须覆盖配送配置、运费规则、人工履约或 tracking 中被触碰的路径。
- 覆盖率阈值在现有测试框架和基线明确后设置，先从核心模块覆盖率开始。
- 未通过测试的功能不交付；无法测试的部分必须说明原因和残余风险。

## 验证工作流

基础构建检查：

```powershell
npm run build --workspace=@dtc/backend
npm run build --workspace=@dtc/storefront
```

本地开发验证：

```powershell
npm run db:up
npm run backend:dev
npm run storefront:dev
npm run demo:proxy
```

测试命令：

```powershell
npm run test
npm run test:demo-customer-login
npm run test:admin-login
```

交付前至少执行相关 workspace test；如果全量 test 可用，执行 `npm run test`。如果测试命令本身缺失或失败，必须先修正测试入口或在交付说明中明确阻塞原因。

## 文档同步规则

以下情况发生时，必须同步更新项目文档：

- 新增前台页面、模板、脚本或目录结构；
- 变更启动命令、运行方式或环境变量；
- 调整支付、物流、履约、测试或部署策略；
- 调整 Lark 同步脚本或导入方式；
- 修改产品范围、业务规则或验收标准；
- 新增需要长期保留的技术决策。

## Git 工作流

- 每完成一个稳定功能切片或里程碑节点后提交。
- 不要把无关主题混在同一个 commit 中。
- commit message 保持简洁明确。
- 不要提交 `.env`、日志、`node_modules`、`.next`、`.medusa` 和本地产物。

## Lark 同步规则

- 普通段落型 Markdown 可使用 `npm run docs:sync:lark` 直接覆盖目标文档。
- 含表格、需要保留更强排版的文档，应优先使用 `npm run docs:import:lark` 导入为新的 Lark 文档。
- 如果 Lark 文档由 Bot 创建，通常也由 Bot 持有文档级编辑权限；如需人工编辑，应单独处理协作者授权。
- 涉及 Lark 输出的文档说明仍需保持中文。

## 当前默认原则

- 生产目标优先，测试环境标准是最低线。
- 默认面向日本市场。
- 移动端必须保持基本可用，不能只满足桌面端。
- 当前可保留模拟支付和模拟物流作为本地 fallback，但 staging 必须朝真实支付 sandbox 和可追踪物流履约推进。
- 文档说明默认使用中文。
- 改功能必须同步补测，交付前必须跑测试。
