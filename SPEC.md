# Popink Market 规格说明书

## 0. 工作流联动

- 需求范围、业务规则和验收标准以本文档为准。
- 阶段拆分、执行顺序和进度记录见 `docs/MILESTONES.md`。
- 开发、文档、验收和 Git 工作流规则见 `docs/WORKFLOW.md`。
- 如果实现过程中发现需求变化，应先更新本文档或在 milestone 的 open decisions 中记录，再进入代码实现。

## 1. 项目背景

Popink Market 是一个面向日本市场纹身行业工作者的 B2B 电商网站。网站主要销售纹身相关产品，包括纹身耗材、设备、色料和护理产品。

项目当前处于早期 demo 阶段，目标不是一次性完成完整商业系统，而是先搭建一个可本地运行、可演示、可继续扩展的电商基础框架。后续根据业务验证结果，再逐步完善前端体验、支付、物流、客户审核、库存和运营能力。

## 2. 项目目标

### 2.1 当前阶段目标

- 基于成熟开源电商框架搭建一个 PC 网页端电商 demo。
- 使用 Medusa.js 作为优先尝试的电商后端框架。
- 支持商品展示、商品分类、购物车、基础下单和后台管理。
- 支持本地部署，方便开发、测试和演示。
- 保留后续部署到轻量级服务器和专业服务器的扩展空间。
- 保留后续向移动端网页、独立前端或 App/小程序迁移的能力。

### 2.2 长期目标

- 支持日本市场的 B2B 电商运营。
- 支持多样化、国际化的支付方式接入。
- 支持更完整的商品规格、库存、优惠、邮件、内容管理和客户管理。
- 支持从本地部署平滑迁移到云服务器或专业生产环境。
- 支持前后端解耦，降低后期改版和多端扩展成本。

## 3. 目标用户

### 3.1 主要用户

- 日本纹身店经营者
- 纹身师
- 纹身行业相关采购人员

### 3.2 管理用户

- 网站运营人员
- 商品管理人员
- 订单处理人员

## 4. 一期 Demo 范围

一期 demo 以验证系统框架、核心电商流程和后台管理能力为主。前端视觉和交互可以保持基础可用，不追求完整商业级设计。

### 4.1 前台功能

- 商品分类浏览
- 商品列表
- 商品详情
- 示例商品数据
- 明示商品价格
- 购物车
- 基础用户注册
- 用户登录
- 下单时填写收货地址
- 下单时填写基础支付信息或选择模拟支付方式
- 提交订单
- 订单基础状态展示

### 4.2 后台功能

- 商品管理
- 分类管理
- 订单查看
- 客户管理
- 库存管理
- 优惠码管理
- 内容页管理
- 邮件通知基础配置

### 4.3 示例商品

一期 demo 可以使用 sample 商品，不要求真实完整 SKU 数据。商品规格可以先保持简单，后续由业务合伙人根据真实商品体系补充。

建议 sample 商品覆盖以下类别：

- 纹身针或针类耗材
- 纹身色料
- 纹身机器或设备
- 护理产品

## 6. 业务规则

### 6.1 商品价格

一期 demo 中商品价格直接公开展示，游客和登录用户均可查看价格。

### 6.2 用户注册

一期 demo 中用户注册只需要基础信息：

- 邮箱
- 用户 ID 或用户名
- 密码

客户审核流程暂不实现。

### 6.3 下单信息

用户下单时需要填写或选择：

- 收货人信息
- 收货地址
- 联系方式
- 基础支付信息或模拟支付方式

### 6.4 支付

一期 demo 可以使用模拟支付或框架提供的基础测试支付能力。当前阶段不接入真实支付网关。

框架必须支持后续接入多样化和国际化支付方式，例如信用卡、Stripe、PayPal 或日本本地支付方式。

### 6.5 物流

一期 demo 不接入真实物流服务。订单中只需要记录配送相关信息，后台可以查看订单配送信息。

## 7. 技术选型方向

### 7.1 优先框架

一期优先尝试 Medusa.js。

选择 Medusa.js 的原因：

- 开源电商框架，适合作为可扩展电商后端基础。
- Headless 架构，前后端解耦，方便后续替换或扩展前端。
- 适合后续接入独立 PC 前端、移动端网页或其他客户端。
- 支持插件和自定义业务逻辑，适合逐步扩展支付、物流、客户、订单等能力。
- 相比 Magento 等重型系统，更适合 demo 阶段快速搭建。

### 7.2 前端原则

- 当前阶段以前端基础可用为主。
- 优先保证商品浏览、购物车和下单流程清晰。
- 页面布局以 PC 网页端为主。
- 组件和接口设计需要保留响应式和移动端迁移空间。

### 7.3 部署原则

部署阶段分为三步：

1. 本地开发和演示部署
2. 轻量级服务器部署
3. 专业服务器或云基础设施部署

一期需要优先保证本地部署清晰可复现。

## 8. 非功能要求

### 8.1 可维护性

- 项目结构需要清晰。
- 配置、环境变量和部署说明需要文档化。
- 业务逻辑尽量通过框架推荐方式扩展，减少对核心框架的侵入式修改。

### 8.2 可扩展性

- 支付、物流、商品规格、客户审核、价格策略等能力需要保留扩展空间。
- 前后端需要尽量解耦。
- 数据模型设计应避免只适配 demo 数据。

### 8.3 可移植性

- 本地环境、轻量级服务器和生产服务器之间应尽量保持一致的启动方式。
- 推荐使用环境变量区分不同环境配置。
- 后续可考虑 Docker 化部署。

### 8.4 国际化准备

目标运营地区为日本。一期 demo 不强制实现完整多语言和多币种，但系统选型和数据设计需要考虑：

- 日本地址格式
- 日元价格展示
- 日本消费税或税务扩展
- 日本本地支付方式扩展
- 后续日文前端内容

## 9. 待确认事项

以下问题需要在后续阶段继续确认：

- 最终商品分类体系
- 真实商品 SKU 和规格模型
- 是否需要日文作为默认语言
- 是否需要后台也支持日文
- 是否需要企业客户审核
- 是否需要登录后查看价格
- 是否需要真实支付以及优先接入哪种支付方式
- 是否需要真实物流以及优先接入哪家物流服务
- 是否需要内容页，例如品牌介绍、使用说明、批发说明、退换货政策
- 是否需要邮件服务商，例如 SendGrid、Mailgun、Amazon SES
- 是否需要 Docker 作为标准部署方式

## 10. 一期验收标准

一期 demo 达到以下条件即可认为完成：

- 可以在本地启动 Medusa.js 后端。
- 可以访问基础前台页面。
- 可以看到 sample 商品。
- 可以浏览商品分类和商品详情。
- 可以将商品加入购物车。
- 可以完成基础注册和登录。
- 可以提交 demo 订单。
- 后台可以查看商品、客户和订单。
- 有清晰的本地启动说明。
- 代码和配置方式不阻碍后续部署到轻量级服务器。
