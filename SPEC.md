# Popink Market 规格说明

## 0. 工作联动

- 需求范围、业务规则、测试环境标准、生产前阻塞项和验收标准以本文件为准。
- 阶段拆分、进度和日志记录见 `docs/MILESTONES.md`。
- 文档、实现、测试和交付规则见 `docs/WORKFLOW.md`。
- 本项目中的说明文档统一使用中文；只有代码、命令、路径、环境变量和第三方固定名称可以保留英文。

## 1. 项目背景

Popink Market 是一个面向日本纹身行业工作者的 B2B 电商项目，目标客户包括纹身店经营者、纹身师与相关采购人员。项目始终以上线生产为目标推进，当前最低标准是形成可部署到 `staging / 测试环境` 的阶段性版本。

当前阶段优先验证：

- Medusa + Next.js 电商骨架能否稳定支撑测试环境；
- 前台、后台、配置、部署和测试能否形成可交付闭环；
- 商品浏览、购物车、结算、订单、支付 sandbox 和物流履约能否按生产路线推进；
- 后续 B2B 客户审核、价格策略、真实物流、真实支付和生产部署是否具备清晰扩展路径。

## 2. 当前阶段目标

### 2.1 测试环境目标

- 建立可本地开发、可测试环境部署、可迁移生产的 Medusa.js + Next.js + PostgreSQL + Redis 项目基线。
- 支持前台商品浏览、商品详情、购物车、客户注册/登录、结算和订单提交。
- 支持后台查看商品、客户、库存和订单。
- 将前台视觉与文案从官方 starter 转为 Popink Market 业务站点风格。
- 补齐基础静态内容页，使站点具备进入测试环境的基本信息结构。
- 结构稳定后立即推进支付和物流模块，不再作为远期预留项。

### 2.2 生产方向

- 面向日本市场继续深化商品、价格、税务和地址规则。
- 为 B2B 客户审核、登录后价格、企业资料、批发价和询价等能力预留路线。
- 支付第一阶段接真实服务 sandbox，本地允许保留模拟支付 fallback。
- 物流第一阶段建立配送配置、运费规则、人工履约和 tracking 记录，再接真实物流商 API。
- 本地、测试环境和生产环境应保持一致的配置习惯、启动结构和验证流程。

## 5. 当前不进入一期的内容

- 生产支付正式收款
- 物流商 API 面单和自动追踪
- 客户审核流程
- 登录后价格展示
- 企业分级价格
- 询价单
- 批量采购
- ERP / 会计系统集成
- 多仓储复杂库存

这些内容不进入一期并不代表放弃生产目标；它们是生产前或后续版本需要明确排期的能力。

## 6. 业务规则

### 6.1 市场与币种

- 当前默认市场为日本。
- 当前默认 region 为 `jp`。
- 当前默认币种为 `JPY`。

### 6.2 商品信息

- 商品名称、handle、SKU 和部分技术标识目前允许保留英文。
- 前台业务文案和项目说明文档应以中文为主，后续再决定是否日文优先或双语。
- 当前 sample 商品用于测试环境验证，后续必须替换为真实商品资料或更接近真实业务的数据。

### 6.3 客户注册与下单

- 当前阶段允许客户直接注册和登录。
- 当前阶段不引入审核制，但后续 B2B 客户审核必须作为扩展路线保留。
- 下单时需要具备基础收货信息、地址、配送和支付方式。

### 6.4 支付

- 本地开发可保留模拟支付 fallback。
- `staging / 测试环境` 默认应接真实支付服务 sandbox。
- 支付供应商在账号、资质、测试密钥明确后锁定。
- 生产上线前必须替换测试密钥，关闭不应暴露的模拟支付入口，并完成失败、取消、回调和幂等处理验证。

### 6.5 物流

- 第一阶段先建立物流抽象、配送配置、运费规则、人工履约和 tracking 记录。
- 暂不强依赖物流商 API。
- 供应商确定后再接真实面单、追踪和状态同步接口。
- 生产上线前必须明确配送范围、费用规则、退换货规则和履约责任边界。

## 7. 技术方向

### 7.1 技术选型

- 后端优先使用 Medusa.js。
- 前台使用官方 Next.js storefront 进行业务化改造。
- 数据库使用 PostgreSQL。
- 缓存使用 Redis。
- 本地依赖通过 Docker Compose 管理。

### 7.2 前端改造原则

- 以测试环境可交付为最低标准，不以单机可访问作为目标终点。
- 首页和导航优先支持商品发现与业务定位展示。
- 移动端必须保持基本可用，不能只满足桌面端截图。

### 7.3 文档与 Lark 同步原则

- 仓库内文档必须以中文为主。
- 需要推送到 Lark 的普通段落文档可使用块写入同步。
- 需要保留表格效果的文档应优先使用 HTML 导入同步。
- 与 Lark 同步相关的脚本和说明也必须保持中文文档化。

## 9. 非功能要求

### 9.1 可维护性

- 文档、代码结构和脚本入口必须清晰。
- 当需求、进度、运行方式、支付物流策略、测试门禁或同步方式变化时，相关文档必须同步更新。
- 实现应尽量遵循 Medusa 和 Next.js 现有模式，避免不必要的侵入式重构。

### 9.2 可扩展性

- 需为后续商品深化、B2B 规则、真实支付和真实物流接入保留扩展空间。
- 前后端应尽量保持解耦。
- Lark 文档同步应同时支持简单覆盖和保留表格的导入两种路径。

### 9.3 可移植性

- 本地、测试环境和生产环境应尽量保持一致的运行习惯。
- 配置应通过环境变量区分。
- 数据重置、启动、构建和测试路径应可重复执行。

### 9.4 测试门禁

- 采用“关键模块 + 覆盖率”标准。
- 核心业务模块必须逐步补齐单元测试：前台购买链路、账号注册登录、配置/env、支付、物流、文档同步脚本。
- 新增功能必须随代码一起提交单元测试。
- 交付前必须运行相关 workspace test；全量 test 可用时运行 `npm run test`。
- 未通过测试的功能不交付；无法测试的部分必须说明原因和残余风险。
- 覆盖率阈值在现有测试框架和基线明确后设置，先从核心模块覆盖率开始。

## 10. 测试环境最低标准

- 使用非开发态启动方式或明确等价的 staging 运行方式。
- 统一域名可访问前台、Store API、Admin、上传资源和健康检查。
- `local / staging / production` 环境变量分层清晰。
- 支付在 staging 默认走真实服务 sandbox。
- 物流在 staging 至少支持配送配置、运费规则、人工履约和 tracking 记录。
- 具备日志、健康检查、进程守护、密钥替换和备份说明。
- 核心功能测试通过。

## 11. 生产前阻塞项

- 替换所有测试密钥、默认管理员密码、JWT secret、cookie secret、数据库凭据。
- 明确 CORS、allowed hosts、Admin 访问范围和公网暴露策略。
- 支付完成 sandbox 流程验证，并明确上线供应商、回调、失败处理和退款策略。
- 物流完成履约流程验证，并明确配送范围、费用、tracking 和退换货规则。
- 完成核心模块单元测试和必要集成验证。
- 明确日志、监控、备份、恢复和数据迁移方案。

## 12. 待确认事项

- 前台正式默认语言是中文、日文还是双语。
- 商品名称和分类名称后续是否需要日文优先。
- Popink Market 是否作为正式品牌名。
- 是否需要真实 logo、品牌色和字体方案。
- 商品图片是继续占位，还是准备真实素材。
- 日本消费税展示方式。
- 测试环境和生产环境目标平台与预算。
- 优先接入的支付供应商和物流供应商。
- 是否需要真实邮件服务。

## 13. 一期验收标准

- 本地可以成功启动 PostgreSQL、Redis、Medusa 后端和前台，用于开发验证。
- 测试环境路线清晰，具备统一入口、环境变量分层和部署说明。
- 前台可访问 `/jp` 和 `/jp/store`。
- 可浏览 sample 商品、查看详情、加入购物车并进入 checkout。
- 可完成客户注册/登录与订单提交。
- 支付路线明确，staging 默认应接真实支付服务 sandbox。
- 物流路线明确，至少支持人工履约和 tracking 信息记录。
- 后台可查看商品、客户和订单。
- 前台具备基础静态内容页。
- `README.md`、`SPEC.md`、`docs/` 中的项目说明为中文且与当前实现一致。
- 关键模块测试门禁已写入工作流，新增功能必须带单元测试。
- Lark 同步脚本可用于输出 `SPEC.md` 与 `docs/MILESTONES.md`。

## 14. 后续重点

### 14.1 结构和环境收口

- 收口启动方式、环境变量分层、部署模板和测试门禁。
- 将开发态入口与测试环境入口明确区分。
- 明确 staging 验收清单。

### 14.2 支付模块

- 选择首个真实支付服务 sandbox。
- 建立支付配置、测试密钥管理、成功/失败/取消流程和回调处理计划。
- 保留本地模拟支付 fallback，但 staging 不以模拟支付作为最终验收。

### 14.3 物流模块

- 建立物流抽象、配送配置、运费规则、人工履约和 tracking 记录。
- 后续根据供应商接入真实面单、追踪和状态同步。

### 14.4 测试覆盖

- 梳理现有核心模块测试缺口。
- 先补关键路径单元测试，再逐步建立覆盖率阈值。
